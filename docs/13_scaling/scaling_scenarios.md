## Сценарий A: 10 школ (~500 учеников)

**Принцип:** максимальная простота. Один сервер, один процесс, одна база. Всё умещается в голове одного разработчика.

### Инфраструктура

| Компонент | Конфигурация | Роль |
|-----------|--------------|------|
| VPS | 4 vCPU, 8 GB RAM | App server + PostgreSQL |
| PostgreSQL | На том же VPS | Основная БД |
| Хранилище | SSD 80–100 GB | Данные + бэкапы |
| CDN | Опционально | Статические ассеты |

**Ориентировочная стоимость:** $30–50 / месяц.

### Узкие места

На этом масштабе явных узких мест нет. 500 учеников не создают нагрузки, способной перегрузить современный VPS.

### Что заложить с самого начала

Эти решения дёшево стоят сейчас и дорого обходятся при их отсутствии позже.

**Row-Level Security (RLS)** — изоляция данных школ на уровне PostgreSQL. Добавить при создании схемы; переделка на 100 школах потребует миграции всех данных и рефакторинга запросов.

**Индексы** — минимальный необходимый набор с первого дня:
```sql
CREATE INDEX idx_answers_student_id   ON answers(student_id);
CREATE INDEX idx_answers_test_id      ON answers(test_id);
CREATE INDEX idx_answers_school_id    ON answers(school_id);
CREATE INDEX idx_tests_school_id      ON tests(school_id);
CREATE INDEX idx_students_school_id   ON students(school_id);
```

**Партиционирование таблицы `answers` по месяцам** — таблица `answers` растёт быстрее всех остальных. Партиционирование по `created_at` (RANGE, monthly) позволит безболезненно архивировать старые данные и поддерживать скорость запросов при росте до сценария C.

**Structured logs** — логировать в JSON с самого начала (`school_id`, `user_id`, `action`, `duration_ms`). Переход с plain text на structured logs на 100 школах — это недели работы и потеря исторических данных для анализа.

## Сценарий B: 100 школ (~10 000 учеников)

**Принцип:** разделить нагрузку чтения и записи; вынести тяжёлые операции из основного потока запросов.

### Инфраструктура

| Компонент | Конфигурация | Роль |
|-----------|--------------|------|
| App Server × 2 | 2–4 vCPU, 4–8 GB RAM | Stateless application layer |
| Load Balancer | Nginx / managed LB | Распределение трафика |
| PostgreSQL Primary | 4–8 vCPU, 16 GB RAM | Запись + транзакции |
| PostgreSQL Read Replica | 4 vCPU, 8 GB RAM | Аналитика + отчёты |
| Redis | 1–2 GB RAM | Кэш сессий, очередь задач |
| CDN | — | Статика, JS/CSS бандлы |
| Object Storage | S3-compatible | PDF-отчёты, медиафайлы |

**Ориентировочная стоимость:** $200–300 / месяц.

### Узкие места и решения

**Генерация PDF-отчётов.** Синхронная генерация PDF блокирует воркеры и деградирует latency для всех пользователей в момент пиковых запросов.

Решение: все запросы на генерацию отчётов уходят в **очередь задач** (Celery + Redis / pg-based queue). Пользователь получает `task_id` и статус «в очереди»; готовый PDF загружается в object storage и отдаётся по ссылке. SLA на генерацию: < 30 секунд с момента постановки в очередь.

**Аналитические запросы.** Тяжёлые агрегации (метрики класса, школы, РОНО) конкурируют с OLTP-трафиком на одном primary.

Решение: все `SELECT`-запросы для дашбордов и аналитики маршрутизируются на **read replica**. Primary принимает только `INSERT` / `UPDATE` / `DELETE`.

**ClickHouse (опционально).** Если аналитические запросы на read replica стабильно превышают 2–3 секунды — добавить ClickHouse как аналитическое хранилище. Данные реплицируются из PostgreSQL через CDC (Debezium / pg_logical). Переход не обязателен на старте сценария B; решение принимается по метрикам.

### Конфигурация Redis

```
Сессии:        TTL 24h,  ~1 KB/сессия
Кэш дашборда:  TTL 60s,  ~50 KB/запись
Очередь PDF:   без TTL,  приоритетная очередь
Rate limiting: TTL 1h,   sliding window
```

## Сценарий C: 1 000 школ (~150 000 учеников)

**Принцип:** горизонтальное масштабирование с изоляцией по `school_id`; автоматическое управление ёмкостью.

### Инфраструктура

| Компонент | Конфигурация | Роль |
|-----------|--------------|------|
| App Servers | Auto-scaling group (min 3, max N) | Stateless application layer |
| Load Balancer | Managed, с health checks | L7 routing |
| PostgreSQL Shards | 4–8 шардов, каждый Primary + Replica | Шардирование по `school_id` |
| PgBouncer | Per-shard | Connection pooling |
| Redis Cluster | 3+ нод | Кэш, очереди, pub/sub |
| ClickHouse | 2–3 нод | Аналитика и агрегации |
| CDN | Global PoPs | Статика + edge caching |
| Object Storage | — | PDF, медиа, бэкапы |
| Kubernetes / managed PaaS | DigitalOcean App Platform, Railway, или облако РУз | Оркестрация |

**Ориентировочная стоимость:** $1 500–2 000 / месяц.

### Узкие места и решения

**Hot-spot школы.** Топ-100 школ по количеству учеников генерируют ~60% всей нагрузки. На едином primary это создаёт неравномерную нагрузку.

Решение: **шардирование по `school_id`**. Схема распределения:
```
school_id % N_shards → номер шарда
```
Крупные школы (топ-100 по активности) при необходимости выносятся на выделенные шарды вручную (dedicated shard).

**Пиковая нагрузка 08:00–09:00.** Одновременный вход 5 000+ сессий за 15 минут.

Решение: **автоскейлинг** app-серверов по метрике CPU/RPS с pre-warming — за 10 минут до прогнозируемого пика добавляются дополнительные инстансы. **PgBouncer** перед каждым шардом ограничивает количество реальных подключений к PostgreSQL (transaction pooling mode).

**Connection exhaustion.** 150 000 учеников × N подключений = исчерпание `max_connections` PostgreSQL.

Решение: PgBouncer в режиме `transaction pooling` снижает количество реальных подключений к БД в 10–20×.

### Стратегия шардирования

```
Шард 0:  school_id 0–249
Шард 1:  school_id 250–499
Шард 2:  school_id 500–749
Шард 3:  school_id 750–999
...
```

Cross-shard запросы (например, рейтинг РОНО) выполняются через ClickHouse, куда данные реплицируются асинхронно. Не допускается выполнение cross-shard JOIN на уровне PostgreSQL.

## Триггеры перехода

### A → B: когда переходить

| Сигнал | Порог | Действие |
|--------|-------|----------|
| Время аналитических запросов | > 2 сек стабильно | Добавить read replica |
| Генерация PDF влияет на latency API | P95 API > 1.5 сек в пиковые часы | Вынести PDF в очередь |
| Количество подключённых школ | 80+ | Начать планирование перехода |
| CPU VPS в пиковые часы | > 70% sustained | Горизонтальный scale-out |

### B → C: когда переходить

| Сигнал | Порог | Действие |
|--------|-------|----------|
| Primary PostgreSQL не справляется с записью | Write latency P95 > 500 мс | Шардирование |
| Read replica отстаёт от primary | Replication lag > 5 сек | Добавить шард / replica |
| Количество подключённых школ | 800+ | Начать планирование перехода |
| Одновременных сессий в пике | > 4 000 | Автоскейлинг + PgBouncer |

## Что НЕ делать

### ❌ Kubernetes до команды 5+ разработчиков

Kubernetes решает проблемы масштабирования, которых на сценариях A и B ещё нет, но добавляет операционную сложность, которая есть всегда. Стоимость обслуживания кластера K8s — это фактически половина ставки DevOps-инженера. До достижения сценария C и формирования полноценной инженерной команды использовать Docker Compose (сценарий A) и managed PaaS / Docker Swarm (сценарий B).

### ❌ Микросервисы без явного bottleneck

Монолит с чёткими модульными границами масштабируется до 1 000+ школ без декомпозиции на сервисы. Преждевременный переход на микросервисы удваивает сложность деплоя, добавляет сетевые задержки между сервисами, усложняет транзакционность и отладку. Переходить к микросервисам только тогда, когда конкретный модуль (например, генерация PDF или аналитика) становится явным bottleneck и требует независимого масштабирования.

### ❌ NoSQL «потому что модно»

Данные Zedly — реляционные по природе: школы, классы, ученики, тесты, ответы — связаны между собой внешними ключами, транзакциями и аналитическими агрегациями. PostgreSQL с правильными индексами, партиционированием и read replica справляется с нагрузкой сценария C (150 000 учеников) без замены на NoSQL. MongoDB, Cassandra, DynamoDB решают другие задачи — горизонтальная запись неструктурированных данных без JOIN. Zedly не имеет этих задач до масштаба, при котором NoSQL начинает оправдывать свою операционную сложность.
