# Product Overview: Zedly

## 1. Что такое Zedly
Zedly — SaaS-платформа онлайн-тестирования и образовательной аналитики для школ Узбекистана. Продукт объединяет подготовку к НТТ, проверку знаний по школьной программе и управленческую аналитику от уровня ученика до уровня РОНО в одном окне. Система запускается через Telegram и web-интерфейс, а в регионах с нестабильным интернетом продолжает работу за счет offline-first сценариев с последующей синхронизацией.

## 2. Для кого
| Роль | Что делает в системе | Что получает на выходе |
|---|---|---|
| Ученик | Проходит тренировочные и контрольные тесты, решает НТТ-симуляции по таймеру | Личный прогресс по предметам и темам, список слабых тем, динамика баллов |
| Учитель | Создает тесты вручную или через AI, назначает классу, проверяет результаты | Автопроверка, аналитика по классу, рейтинг сложных тем для повторения |
| Директор | Контролирует учебные результаты по классам и предметам | KPI школы, проблемные классы, сравнение параллелей, отчеты для управленческих решений |
| Родитель | Следит за результатами ребенка и пропусками тестов | Понятный прогресс ребенка, уведомления о снижении результатов, рекомендации по фокусу |
| РОНО (район) | Сравнивает школы района и контролирует качество подготовки | Сводная районная аналитика, школы риска, тренды успеваемости по предметам |
| Минпрос | Получает агрегированную картину по регионам | Иерархическая отчетность без ручной консолидации, база для управленческих программ |

## 3. Ключевые возможности
- Единая аналитическая вертикаль: одна модель данных покрывает 6 ролей (ученик, учитель, директор, родитель, РОНО, Минпрос) без дублирования отчетов в разных системах.
- НТТ-симулятор: формат, тайминг и структура попытки приближены к реальному национальному тесту; после завершения формируется детальный разбор по темам.
- Telegram-нативные сценарии: учитель назначает тест и отслеживает результаты в привычном канале, ученик получает задания и уведомления без установки нового сложного ПО.
- Offline-first режим: при пропадании сети попытка и ответы сохраняются локально, после восстановления связи отправляются на сервер с защитой от дублей.
- AI-генерация тестов из узбекских учебников: система предлагает набор вопросов по теме и уровню сложности, учитель утверждает перед публикацией.
- Мультитенантность школ: каждая запись привязана к `school_id`, а доступ ограничивается политиками RLS, что исключает просмотр данных другой школы.
- Автоматическая проверка и моментальная обратная связь: закрытые задания проверяются сразу, ученик и учитель получают результат без ручной обработки.
- Управленческие отчеты: директор, РОНО и Минпрос получают агрегированные срезы по предметам, классам и периодам для быстрых решений.

## 4. Чем отличается от конкурентов
| Критерий | Zedly | Базовые тестовые инструменты |
|---|---|---|
| Аналитика по вертикали «ученик → школа → РОНО» | Да, единая сквозная аналитика в одном контуре | Обычно только уровень теста/класса без районной иерархии |
| Подготовка к НТТ | Встроенный НТТ-симулятор с тематическим разбором | Обычно отсутствует или реализована как общий тест без НТТ-логики |
| Telegram-нативность | Полноценные рабочие сценарии для учителей и учеников | Чаще всего отдельный web-кабинет без native Telegram UX |
| Работа при слабом интернете | Offline-first: локальное сохранение + отложенная синхронизация | Как правило, online-only; при обрыве связи попытка теряется |
| AI-генерация из локального контента | Генерация вопросов из узбекских учебников с контролем учителя | Обычно шаблонные банки вопросов без привязки к местной программе |
| Изоляция данных школ | Жесткая мультитенантность через `school_id` и RLS | Нередко слабая сегментация, рассчитанная на один аккаунт/организацию |
| Отчетность для госуправления | Сразу учитывает роли РОНО и Минпрос | Обычно не покрывает государственную иерархию отчетности |

## 5. Технологический стек
- Архитектура: `Modular Monolith` с доменными модулями (auth, test engine, analytics, notifications, integrations) для быстрого MVP и контролируемого роста.
- Основная БД: `PostgreSQL` как источник истины, мультитенантность через `school_id` и политики `Row Level Security`.
- Операционный слой: `Redis` для кэша, rate limiting, фоновых задач и буферизации синхронизации офлайн-событий.
- Канал взаимодействия: `Telegram Bot` как основной интерфейс для ежедневных сценариев учителя и ученика; web-панель для управленческой аналитики.
- Траектория масштабирования: при росте объема аналитических событий подключается `ClickHouse` для тяжелых агрегатов без ломки доменной модели.

## 6. Статус
Текущая фаза: **MVP**.

В MVP уже входят:
- Создание и назначение тестов.
- Прохождение тестов учеником и автопроверка закрытых вопросов.
- Базовая аналитика для учителя и директора.
- Telegram-нативные ключевые сценарии.
- Изоляция данных по `school_id`.

В текущем цикле развития:
- Расширение отчетности для РОНО и Минпрос.
- Усиление offline-first механики (конфликты синхронизации и повторная отправка).
- Доработка AI-генерации для стабильного качества вопросов по школьной программе.
