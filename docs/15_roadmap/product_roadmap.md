# Product Roadmap: staged prompt for development workflow

## Цель документа
Этот файл — практический «промпт-план» по стадиям разработки продукта. Его можно использовать как шаблон для planning-сессий, постановки задач команде и синхронизации между Product/Design/Backend/Frontend/QA/DevOps.

---

## Как использовать
1. Проходите стадии **по порядку**: от Discovery до Post-Release.
2. На каждой стадии отвечайте на вопросы из блока «Что делаем».
3. Фиксируйте артефакты из блока «Результат стадии».
4. Не переходите дальше, пока не выполнены «Критерии готовности».

---

## Стадия 0. Инициация (Problem Framing)

### Что делаем
- Формулируем проблему пользователя в 1–2 предложениях.
- Определяем сегмент (кто именно страдает от проблемы).
- Описываем бизнес-эффект (зачем это компании).
- Определяем границы: что **входит** и что **не входит** в текущий цикл.

### Входные данные
- Vision и стратегические цели.
- Текущие продуктовые ограничения.

### Результат стадии
- Одностраничный brief:
  - Problem statement
  - Target user
  - Business impact
  - Scope / Out of scope

### Критерии готовности
- Есть согласованный owner инициативы.
- Есть чёткие границы работ.

---

## Стадия 1. Discovery и валидация гипотез

### Что делаем
- Формируем гипотезы ценности и риска.
- Собираем evidence: интервью, аналитика, обратная связь, support-cases.
- Проверяем альтернативы: build / buy / postpone.
- Выделяем топ-риски: продуктовые, технические, правовые, операционные.

### Входные данные
- Problem brief из стадии 0.
- Доступные данные аналитики.

### Результат стадии
- Discovery note:
  - Гипотезы
  - Evidence
  - Решение (идём/не идём)
  - Основные риски и план их снижения

### Критерии готовности
- Есть подтверждённая ценность для пользователя.
- Есть решение по запуску в разработку.

---

## Стадия 2. Product specification (PRD-lite)

### Что делаем
- Формулируем user stories и core-сценарии.
- Определяем функциональные требования (must/should/could).
- Фиксируем acceptance criteria для ключевых сценариев.
- Добавляем KPI/метрики успеха с формулами и baseline.

### Входные данные
- Discovery note.

### Результат стадии
- PRD-lite:
  - User stories
  - Scope v1
  - Acceptance criteria
  - KPI и целевые значения

### Критерии готовности
- Все ключевые сценарии покрыты AC.
- Есть единый словарь терминов/ролей/permission-key/metric-key.

---

## Стадия 3. Solution design (UX + Architecture)

### Что делаем
- Проектируем UX-флоу и состояния интерфейса.
- Определяем API-контракт, модели данных, события и интеграции.
- Выбираем архитектурные решения и фиксируем trade-offs.
- Определяем требования безопасности и соответствия.

### Входные данные
- PRD-lite.

### Результат стадии
- Design package:
  - User flows
  - API draft
  - Data model update
  - Architecture notes
  - Security checklist

### Критерии готовности
- Нет критических конфликтов между UX/API/Data.
- Есть трассировка: scenario -> endpoint -> data -> AC.

---

## Стадия 4. Planning и декомпозиция

### Что делаем
- Разбиваем работу на epics/tasks с оценками.
- Определяем зависимости и критический путь.
- Планируем релизы (MVP -> Beta -> GA).
- Назначаем владельцев и дедлайны.

### Входные данные
- Design package.

### Результат стадии
- Delivery plan:
  - Backlog с приоритетом
  - Milestones
  - Release scope
  - Риск-лог

### Критерии готовности
- Каждая задача имеет owner, estimate, definition of done.
- Согласован план релиза и план отката.

---

## Стадия 5. Implementation

### Что делаем
- Реализуем задачи короткими итерациями.
- Поддерживаем contract-first подход (API/schema first).
- Ведём changelog изменений в спецификациях.
- Проводим code review и проверяем соответствие AC.

### Входные данные
- Delivery plan.

### Результат стадии
- Код + обновлённые спецификации + миграции + feature flags.

### Критерии готовности
- Нет блокирующих дефектов (P0/P1).
- Все обязательные проверки CI проходят.

---

## Стадия 6. Testing и стабилизация

### Что делаем
- Покрываем unit/integration/e2e/acceptance тестами.
- Проверяем негативные сценарии и edge-cases.
- Проводим performance/security/regression проверки.
- Подготавливаем release notes.

### Входные данные
- Реализация из стадии 5.

### Результат стадии
- Test report:
  - Покрытие сценариев
  - Список дефектов
  - Статус исправлений

### Критерии готовности
- Все критичные AC валидированы.
- Риски релиза приняты владельцами.

---

## Стадия 7. Release и rollout

### Что делаем
- Выпускаем в controlled rollout (например, 5% -> 25% -> 100%).
- Мониторим технические и продуктовые метрики.
- Поддерживаем оперативный канал инцидентов.
- Применяем rollback-план при деградации.

### Входные данные
- Test report + release checklist.

### Результат стадии
- Release log и rollout-отчёт.

### Критерии готовности
- Стабильные метрики после выхода на 100%.
- Нет незакрытых критических инцидентов.

---

## Стадия 8. Post-release review

### Что делаем
- Сверяем фактические результаты с KPI.
- Анализируем отклонения и root causes.
- Фиксируем lessons learned.
- Обновляем roadmap следующего цикла.

### Входные данные
- Product/tech analytics + feedback.

### Результат стадии
- Post-release report:
  - KPI факт/план
  - Что сработало / что нет
  - Action items следующего квартала

### Критерии готовности
- Приняты решения по следующему инкременту.
- Обновлён roadmap и backlog.

---

## Универсальный master-prompt (шаблон для команды)

Скопируйте и заполните:

```text
Контекст:
- Инициатива: <название>
- Стадия: <0..8>
- Цель стадии: <что должно быть достигнуто>
- Ограничения: <сроки/ресурсы/техограничения>

Входные артефакты:
- <список документов/ссылок>

Сделай:
1) Перечень конкретных шагов на этой стадии.
2) Список артефактов, которые нужно подготовить.
3) Риски и способы их снизить.
4) Критерии готовности и check-list проверки.
5) Предложение по следующей стадии.

Формат ответа:
- Краткое резюме
- План работ (таблица)
- Риски
- Definition of Done
- Next step
```

---

## Мини-checklist консистентности между стадиями
- Термины ролей и permission-key синхронизированы.
- KPI из PRD совпадают со словарём метрик.
- Каждый endpoint имеет покрытие в acceptance criteria.
- User flows не конфликтуют с жизненным циклом пользователя.
- Data model покрывает все backend-сущности.
