## 1. Роли и их характеристики

| Роль | `role` в JWT | Scope данных | Привязка к школе | Создаётся |
|---|---|---|---|---|
| **Ученик** | `student` | Свои результаты + тесты своего класса | Одна школа, один класс | Создаётся директором школы |
| **Учитель** | `teacher` | Свои тесты + свои классы | Одна школа, N классов | Создаётся директором школы |
| **Директор** | `director` | Вся школа | Одна школа | Создаётся РОНО |
| **Психолог** | `psychologist` | Психологические и профориентационные тесты своей школы | Одна школа | Назначается директором/администратором |
| **Родитель** | `parent` | Один ребёнок | Через ребёнка | Через код ребёнка |
| **РОНО-инспектор** | `inspector` | Агрегат по школам района | Один или несколько районов | Создаётся Министерством |
| **Министерство** | `ministry` | Национальная агрегация | Страна | Создаётся SuperAdmin |
| **SuperAdmin** | `superadmin` | Глобальные операции платформы и агрегаты без PII | Без школы (`school_id = NULL`) | Назначается владельцем платформы |

## 3. Детализация частичных разрешений ⚠️

### 3.1 Учитель

| Действие | Условие разрешения | Условие отказа |
|---|---|---|
| `create_test_ai_generation` | Freemium: ≤ 5 AI-генераций/день. Платный: ≤ 20/день | Дневная квота исчерпана → 429 `AI_GENERATION_DAILY_LIMIT` |
| `view_test_details_with_answers` | Только для своих тестов (`teacher_id = jwt.user_id`) | Чужой тест → 404 (не 403, не раскрывать существование) |
| `edit_any_school_test` | Только директор — и только при `role = director` | Учитель не может редактировать тесты коллег |

| `unpublish_from_marketplace` | Только автор теста (`teacher_id = jwt.user_id`) | Другой учитель (даже из той же школы) → 403 |

| `deactivate_student` | Только учеников своих классов | Ученик чужого класса → 403 |

## 4. Бизнес-правила изоляции данных

### 4.1 Изоляция учителя — «Только свои классы»

**Правило:** Учитель видит только данные классов, в которых он является основным учителем (`classes.teacher_id = jwt.user_id`) или ко-учителем (`class_teachers.teacher_id = jwt.user_id`).

**Что конкретно изолировано:**

```
✅ Учитель ВИДИТ:
   - Список учеников: только своих классов
   - Результаты тестов: только по своим тестам
   - Аналитику: только по своим классам и предметам
   - Сертификаты: только выданные за свои тесты
   - Уведомления: только своих учеников

❌ Учитель НЕ ВИДИТ:
   - Учеников из других классов той же школы
   - Результаты по тестам коллег
   - Аналитику других учителей
   - Общую аналитику школы (только директор)
   - Email/телефоны других учителей школы
```

**Реализация (middleware):**

```typescript
// Проверка при GET /schools/{school_id}/users?role=student
if (jwt.role === 'teacher') {
  const teacherClassIds = await db.query(
    'SELECT id FROM classes WHERE teacher_id = $1 AND school_id = $2',
    [jwt.user_id, jwt.school_id]
  );
  // Добавляем фильтр: только ученики из классов учителя
  filters.class_id_in = teacherClassIds;
}
```

**Граничный случай — ко-учитель:**
Если учитель ведёт несколько предметов в одном классе (например, математику и информатику), он видит всех учеников этого класса, но аналитика фильтруется по предмету его тестов.

### 4.2 Изоляция директора — «Только своя школа»

**Правило:** Директор видит все данные своей школы (`school_id = jwt.school_id`) и ничего за её пределами. Реализуется через RLS в PostgreSQL.

**Что конкретно изолировано:**

```
✅ Директор ВИДИТ:
   - Всех пользователей школы (учителя, ученики, родители)
   - Все тесты всех учителей школы
   - Все результаты всех классов
   - Аналитику по всей школе (на платном плане)
   - Историю платежей и лицензию школы
   - Audit log школы (на платном плане)

❌ Директор НЕ ВИДИТ:
   - Данные других школ района
   - Данные других школ (даже если знает их school_id — RLS блокирует)
   - Детальные данные по ученикам других школ в межшкольных челленджах
     (только агрегат: «Школа №17 — 74%»)
   - Admin panel Zedly
```

**RLS-политика:**

```sql
CREATE POLICY director_school_isolation ON tests
  USING (school_id = current_setting('app.current_school_id')::VARCHAR);
-- Действует для ALL операций (SELECT, INSERT, UPDATE, DELETE)
```

**Граничный случай — директор нескольких школ:**
Если один человек является директором двух школ (например, сеть частных школ), он имеет два отдельных аккаунта с разными `school_id`. Переключение между школами = повторный вход с другим аккаунтом. Единый «директор сети» — Enterprise-функция с ролью `network_director` (в roadmap).

---

### 4.3 Изоляция РОНО-инспектора — «Только свой район»

**Правило:** Инспектор видит агрегированные данные только по школам своих районов. Список районов хранится в `inspector_districts`. Инспектор никогда не видит данные по отдельным ученикам или учителям — только агрегаты.

**Что конкретно изолировано:**

```
✅ Инспектор ВИДИТ:
   - Рейтинг школ своих районов
   - Агрегированную аналитику по предметам и параллелям
   - НТТ-готовность 11 классов (агрегат по школе, не по ученику)
   - Активность учителей (только в разрезе школ, не поимённо)
   - Квартальные отчёты по своим районам

❌ Инспектор НЕ ВИДИТ:
   - Школы других районов
   - Данные по конкретным учителям (ФИО, результаты)
   - Данные по конкретным ученикам (ФИО, оценки)
   - Тесты и вопросы (содержимое)
   - Данные о платежах и лицензиях школ
```

**Уровень агрегации — минимальная единица:**

| Сущность | Что видит инспектор |
|---|---|
| Школа | Название, средний балл, охват, активность |
| Учитель | Только число (14 активных учителей из 20) |
| Ученик | Только число и агрегат (412 учеников, средний балл 74%) |
| Тест | Только число (94 теста за месяц) |
| Результат | Только агрегат по школе/предмету |

**Защита от идентификации по малым группам:**
Если в школе < 5 учеников по предмету — метрика не показывается (NULL), чтобы нельзя было идентифицировать конкретного ученика по баллу.

---

### 4.4 Изоляция родителя — «Только свой ребёнок»

**Правило:** Родитель видит только данные ребёнка, к которому привязан через `parent_profiles.child_user_id`. Один родитель может быть привязан к нескольким детям (например, несколько детей в одной школе).

**Что конкретно изолировано:**

```
✅ Родитель ВИДИТ:
   - Результаты тестов своего ребёнка
   - Список назначенных тестов (название, дедлайн, статус)
   - Сертификаты ребёнка
   - График прогресса ребёнка
   - Уведомления о результатах

❌ Родитель НЕ ВИДИТ:
   - Результаты других учеников (даже одноклассников)
   - Место ребёнка в классе (только если ребёнок сам показал Result Card)
   - Содержимое тестов (вопросы и ответы)
   - Данные учителей школы
   - Аналитику класса или школы
```

**Особый случай — развод/опека:**
Родительский код (`parent_code`) генерируется при создании аккаунта ученика директором. Если доступ нужно ограничить (суд, опека), директор может сбросить `parent_code` и выдать новый только одному из родителей через `POST /admin/students/{id}/reset-parent-code`.

## 5. Edge Cases разрешений

### 5.1 Учитель уволился из школы

**Триггер:** Директор деактивировал аккаунт учителя (`users.status = 'inactive'`).

**Немедленные последствия (в момент деактивации):**

| Объект | Что происходит |
|---|---|
| Сессия учителя | JWT инвалидируется немедленно (добавляется в blacklist Redis) |
| Доступ к платформе | Заблокирован. Попытка использовать refresh token → 401 `ACCOUNT_DISABLED` |
| Активные тесты с открытыми сессиями | Сессии учеников продолжаются. Ученики могут досдать тест до дедлайна |
| Активные назначения | Остаются в силе до дедлайна. Ученики видят тест в Home и могут его пройти |
| Тесты в статусе `draft` | Остаются, но недоступны для назначения |
| Telegram-бот | Уведомления через бот прекращаются (токен инвалидирован) |

**Долгосрочные последствия:**

| Объект | Что происходит | Кто управляет |
|---|---|---|
| Тесты учителя (`status = published`) | Переходят под управление директора. Директор видит их в библиотеке с меткой «Автор: Акбаров Ш. (неактивен)» | Директор |
| Классы учителя | Классы остаются. Директор назначает нового учителя через `PATCH /classes/{id}/teacher_id` | Директор |
| Результаты учеников | Сохраняются навсегда. Аналитика не прерывается | — |
| Сертификаты | Остаются действительными. Учитель-автор указан как «Акбаров Ш.» — не аннулируется | — |
| Marketplace-публикации | Тесты остаются в маркетплейсе. Директор может отозвать от имени школы | Директор |
| Аккаунт учителя | Хранится 90 дней в статусе `inactive`, затем — архивирование (данные остаются, вход невозможен) | Система |

**Реактивация:** Если учитель вернулся — директор меняет `status → active`. Доступ восстанавливается полностью. JWT придётся получить заново (через login).

**Что директор НЕ может сделать:**
- Удалить тесты уволенного учителя, если по ним есть завершённые сессии учеников (защита истории)
- Изменить авторство тестов (учитель остаётся автором навсегда)
- Получить доступ к личному Telegram-аккаунту учителя

### 5.2 Школа отменила лицензию — Graceful Downgrade

**Сценарий А: Платный план истёк, школа перешла на Freemium**

Переход происходит в 00:00 дня после истечения `plan_expires_at` (по расписанию cron-job).

| Функция | До истечения (платный) | После истечения (Freemium) | Данные |
|---|---|---|---|
| Число активных учителей | Без лимита (Enterprise) / до 25 (Standard) | Первые 5 учителей по `last_active_at` остаются активными, остальные → `status: inactive` | Сохраняются |
| Число учеников | Без лимита | 150 учеников на школу. Если было 800 → 650 получают `status: inactive` (по принципу «последние зарегистрированные» деактивируются первыми) | Сохраняются |
| School Analytics (дашборд директора) | ✅ | ❌ → 403 `PLAN_UPGRADE_REQUIRED` | Сохраняются |
| Audit Log (school-wide) | ✅ | ❌ | Сохраняются |
| AI-генерация | 20/день | 5/день | — |
| Отчёты PDF | ✅ | Только class-level отчёты | — |
| Исторические данные | Доступны | Доступны (не удаляются) | ✅ |
| Тесты и результаты | Доступны | Доступны | ✅ |

**Уведомления о downgrade:**
- За 30 дней до истечения: email + Telegram директору
- За 7 дней: повторное уведомление
- За 1 день: финальное предупреждение
- В момент downgrade: уведомление директору + деактивированным учителям (с объяснением)

**Реактивация после downgrade:**
Директор оплачивает новый план → немедленная реактивация всех деактивированных пользователей (статус `active`) в течение 5 минут после подтверждения оплаты.

---

**Сценарий Б: Школа приостановлена (неоплата 90+ дней)**

| Фаза | Дней после истечения | Что происходит |
|---|---|---|
| Grace period | 0–7 | Freemium-режим (см. Сценарий А) |
| Warning | 8–30 | Ежедневные уведомления директору |
| Soft suspension | 31–90 | Новые тесты запрещены. Ученики могут досдать назначенные тесты. Вход разрешён только директору и для просмотра данных |
| Hard suspension | 91+ | Вход всех пользователей заблокирован. Данные сохраняются |
| Data retention | 91–455 (1 год) | Данные хранятся на cold storage (R2 Glacier). Возможна реактивация |
| Deletion | 455+ | Данные удаляются необратимо. За 30 дней до удаления — финальное уведомление |

**При реактивации после hard suspension:**
Все данные восстанавливаются из cold storage в течение 24 часов. Пользователи получают уведомление.

### 5.3 Ученик переходит в другую школу

**Триггер:** Учитель или директор выполняет перевод ученика (`left_reason = 'transferred'` в `class_students`).

**Модель данных при переводе:**

```
Школа А (исходная)                   Школа Б (новая)
├── class_students                    ├── class_students
│   └── [student_id, left_at=NOW(),   │   └── [student_id, enrolled_at=NOW()]
│         left_reason='transferred']  │
├── test_sessions (school_id=A)       ├── test_sessions (school_id=B)
│   └── [сохраняются навсегда]        │   └── [новые с нуля]
├── answers (school_id=A)             ├── answers (school_id=B)
│   └── [сохраняются навсегда]        │   └── [новые с нуля]
└── certificates                      └── certificates
    └── [остаются у ученика]              └── [новые]
```

**Что следует за учеником в Школу Б:**

| Данные | Переходят? | Примечание |
|---|---|---|
| Аккаунт (`users` запись) | ✅ | `user_id` неизменен, меняется `school_id` |
| Логин / пароль / Telegram | ✅ | Без изменений |
| Сертификаты | ✅ | Исторические сертификаты остаются в портфолио |
| Портфолио (вкладка «Сертификаты») | ✅ | Полная история |
| НТТ-симуляции | ✅ | История НТТ сохраняется |
| Результаты тестов | ⚠️ | Видны ученику в портфолио, но НЕ видны учителям Школы Б |

**Что НЕ переходит в Школу Б:**

| Данные | Почему |
|---|---|
| `test_sessions` и `answers` Школы А | `school_id` остаётся A. RLS Школы Б не даёт доступ |
| Аналитика Школы А | Снапшоты привязаны к `school_id = A` |
| Место в рейтинге класса Школы А | Исторические позиции не отображаются в новой школе |
| Незавершённые сессии Школы А | Ученик не может их завершить (нет назначения от класса Школы Б) |

**Поведение портфолио ученика после перевода:**

```
Вкладка «Результаты»:
  ├── Секция «Школа Б (текущая)» — тесты с сентября 2026
  └── Секция «Школа А (архив)» — результаты до августа 2026
      [Данные видны только ученику. Учителя Школы Б не видят.]

Вкладка «Сертификаты»:
  └── Все сертификаты из обеих школ (сортировка по дате)

Вкладка «НТТ-симуляции»:
  └── Полная история из обеих школ
```

**Особый случай — перевод внутри одной школы (смена класса):**
`school_id` не меняется. `class_id` в `student_profiles` обновляется. Все исторические данные (`test_sessions`, `answers`) остаются видны учителю нового класса через портфолио ученика (тот же `school_id`).

**Регуляторный вопрос — право на удаление данных:**
Ученик (через родителя) может запросить удаление данных в Школе А. Этот запрос обрабатывается через команду Zedly вручную (GDPR-analogous request). Сертификаты при этом НЕ удаляются — они являются официальными документами.

## 6. Реализация на уровне middleware

### 6.1 Порядок проверок при каждом запросе

```
HTTP Request
    │
    ▼
[1] Authentication middleware
    ├── Проверить JWT: подпись, TTL, blacklist (Redis bl:{jti})
    ├── Извлечь: user_id, role, school_id из payload
    └── 401 если невалидный → stop

    ▼
[2] Rate limit middleware
    ├── Проверить счётчики в Redis по IP и user_id
    └── 429 если превышен → stop

    ▼
[3] Role authorization middleware
    ├── Сопоставить role из JWT с требуемой ролью эндпоинта
    └── 403 ROLE_FORBIDDEN если роль не разрешена → stop

    ▼
[4] Plan check middleware (только для 🔒-эндпоинтов)
    ├── Проверить schools.subscription_plan
    └── 403 PLAN_UPGRADE_REQUIRED если функция недоступна на плане → stop

    ▼
[5] Resource ownership middleware
    ├── Установить SET LOCAL app.current_school_id (для RLS)
    ├── Установить SET LOCAL app.current_user_id
    ├── Установить SET LOCAL app.current_role
    └── Дальнейшая фильтрация на уровне SQL через RLS

    ▼
[6] Business logic (controller)
    └── Запрос к БД (RLS применяется автоматически)
```

### 6.2 Декоратор эндпоинта (псевдокод)

```typescript
@RequireRoles(['teacher', 'director'])
@RequirePlan(['basic', 'standard', 'enterprise'])
@RateLimit({ window: '1h', max: 60 })
async function getSchoolAnalytics(req: Request) {
  // RLS автоматически ограничивает данные school_id из JWT
  const data = await db.query(
    'SELECT * FROM analytics_snapshots WHERE entity_type = $1',
    ['school']
  );
  return data;
}
```

### 6.3 Коды ошибок авторизации

| Код HTTP | `error.code` | Когда |
|---|---|---|
| 401 | `UNAUTHORIZED` | Нет токена или токен невалидный |
| 401 | `TOKEN_EXPIRED` | Access token истёк |
| 401 | `ACCOUNT_DISABLED` | Аккаунт деактивирован |
| 403 | `ROLE_FORBIDDEN` | Роль не имеет права на действие |
| 403 | `PLAN_UPGRADE_REQUIRED` | Функция недоступна на текущем плане |
| 403 | `SCHOOL_ACCESS_FORBIDDEN` | Попытка доступа к данным другой школы |
| 403 | `DISTRICT_ACCESS_FORBIDDEN` | Инспектор + чужой район |
| 403 | `RESOURCE_OWNERSHIP_REQUIRED` | Ресурс принадлежит другому пользователю |
| 403 | `RATING_REQUIRES_COPY` | Оценка без предварительного копирования |
| 403 | `RETAKE_NOT_ALLOWED` | Повторное прохождение запрещено |
| 403 | `ACTIVE_SESSIONS_EXIST` | Нельзя архивировать тест с активными сессиями |
| 403 | `CANNOT_RATE_OWN_TEST` | Нельзя оценить свой тест |
| 403 | `CANNOT_COPY_OWN_TEST` | Нельзя скопировать свой тест |
