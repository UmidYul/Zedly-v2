# Test Engine — Движок создания и прохождения тестов

**Файл:** `docs/03_features/test_engine.md`  
**Приоритет:** MVP  
**Статус:** Production Blueprint

---

## Цель фичи

Test Engine — ядро платформы Zedly. Обеспечивает полный цикл: создание теста учителем → назначение ученикам → прохождение в любых условиях (онлайн / Telegram / offline) → автоматическая проверка → аналитика результатов.

Без Test Engine платформа не существует. Всё остальное (AI-генерация, аналитика, сертификаты) — надстройки над ним.

---

## Кто использует (роли)

| Роль | Действие |
|------|----------|
| Учитель | Создаёт тест, назначает классу, видит результаты своего класса |
| Ученик | Проходит тест (веб / Telegram / offline) |
| Директор | Видит агрегированные результаты по школе (платный план) |
| РОНО-инспектор | Видит агрегированные результаты по школам района |
| Родитель | Видит результат своего ребёнка (только просмотр) |

---

## Типы вопросов

### 1. Одиночный выбор (MCQ — Multiple Choice Question)
- 1 правильный ответ из 4 вариантов
- Автоматическая проверка
- Баллы: +1 за правильный, 0 за неправильный (без штрафов по умолчанию)
- **НТТ-режим**: возможен штраф −0.25 за неправильный ответ (настройка)

### 2. Множественный выбор
- 2–5 правильных ответов из 5–8 вариантов
- Частичные баллы: N правильных × (1/total_correct)
- Пример: 3 из 4 правильных выбрано → 0.75 балла

### 3. Открытый ответ (текст)
- Ручная проверка учителем (по умолчанию)
- Опциональный AI-автограйдинг (Phase 2) для коротких ответов
- Статус «ожидает проверки» до момента, когда учитель проверил

### 4. Сортировка / соответствие (Matching)
- Два столбца: левый (термины) ↔ правый (определения)
- Drag-and-drop на веб, нумерованный выбор в Telegram
- Полные баллы только при 100% правильных парах (или частичные — по настройке)

### 5. Вопрос с изображением
- Изображение прикрепляется к вопросу или к варианту ответа
- Максимальный размер: **5 MB** на изображение
- Форматы: JPEG, PNG, WebP
- В offline-режиме изображения скачиваются вместе с тестом
- В Telegram: изображение отправляется как photo, ответ — кнопками

## Режимы теста

### 1. Обычный тест (Standard)
- Учитель создаёт тест, назначает классу с дедлайном
- Ученики проходят в любое время до дедлайна
- Можно разрешить 1 попытку или N попыток (учитель выбирает)
- Результаты видны учителю после завершения

### 2. НТТ-симулятор (NTT Mode)
- Формат национального тестирования для поступления в университет
- **60 вопросов**, **90 минут**, строгий таймер
- Таймер виден постоянно, нельзя поставить на паузу
- Принудительное завершение при истечении времени — автосохранение текущих ответов
- Вопросы в фиксированном порядке (нельзя перемешать)
- Разбор ошибок по темам после завершения
- Каждая попытка — отдельная сессия (история сохраняется)

### 3. Межшкольный челлендж (Challenge Mode)
- Директор школы A бросает вызов школе B
- Публичный тест с дедлайном **48 часов**
- Одновременный старт для всех участников
- Результаты публичны на уровне школ (не отдельных учеников)
- После завершения генерируется карточка-результат для обеих школ

### 4. Offline-тест (Offline Mode)
- Учитель публикует тест → ученик видит кнопку «Скачать для прохождения offline»
- PWA Service Worker кэширует: данные теста, изображения, таймер
- Ученик проходит без интернета; ответы хранятся локально (IndexedDB)
- При восстановлении соединения — автоматическая синхронизация
- Если тест истёк по времени: ответы принимаются с меткой `late_submission = true` (учитель видит это)
- Не доступно для НТТ-симулятора (требует строгого контроля)

## User Flow

### Flow A: Учитель создаёт и назначает тест

```
1. Учитель заходит в раздел «Мои тесты»
2. Нажимает «Создать тест»
3. Вводит:
   - Название теста (макс. 120 символов)
   - Предмет (выбор из справочника)
   - Класс (1–11)
   - Режим (Standard / NTT / Challenge / Offline)
4. Добавляет вопросы:
   a. Вручную через интерфейс
   b. Из AI-генератора (отдельная фича)
   c. Из своей библиотеки (сохранённые вопросы)
   d. Из Marketplace (публичные тесты других учителей)
5. Настраивает параметры:
   - Дедлайн (дата + время)
   - Количество попыток (1 по умолчанию)
   - Перемешивать вопросы? (Y/N)
   - Перемешивать варианты? (Y/N)
   - Показать правильные ответы после завершения? (Y/N)
   - Разрешить прохождение offline? (Y/N)
6. Назначает классу(ам) или конкретным ученикам
7. Опционально: отправляет ссылку/уведомление через Telegram-бота
8. Тест активен — ученики могут начать
```

**Время создания теста вручную (5 вопросов MCQ):** ≤ 5 минут  
**Время создания теста через AI-генерацию:** ≤ 2 минут

---

### Flow B: Ученик проходит тест (веб)

```
1. Ученик получает ссылку (Telegram / email / прямой URL)
2. Авторизуется (Telegram Login или email)
3. Видит экран старта: название, количество вопросов, таймер, попытки
4. Нажимает «Начать» — фиксируется время старта (session.started_at)
5. Отвечает на вопросы:
   - MCQ: кнопки выбора
   - Открытый ответ: текстовое поле
   - Matching: drag-and-drop
   - С изображением: см. изображение + выбор ответа
6. Прогресс-бар показывает N из M вопросов
7. Можно вернуться к предыдущим вопросам (если не НТТ-режим)
8. Нажимает «Завершить» или таймер истекает
9. Ответы отправляются на сервер → автоматическая проверка
10. Экран результата: балл, правильные/неправильные, разбор по темам
11. Генерируется карточка-результат (шейрабельная)
```

---

### Flow C: Ученик проходит тест через Telegram-бот

```
1. Учитель нажимает «Отправить в Telegram» — бот рассылает ссылку в чат класса
2. Ученик нажимает на кнопку в боте
3. Бот отправляет вопросы по одному сообщению
4. MCQ: inline-кнопки A/B/C/D
5. Matching: нумерованный выбор «1-B, 2-A, 3-C...»
6. Открытый ответ: текстовое сообщение
7. После каждого ответа бот подтверждает и отправляет следующий вопрос
8. По завершении: бот отправляет результат и карточку-результат
```

## Зависимости от данных (таблицы)

| Таблица | Описание | Ключевые поля |
|---------|----------|---------------|
| `tests` | Метаданные теста | `id`, `school_id`, `teacher_id`, `subject_id`, `mode`, `time_limit`, `created_at` |
| `questions` | Вопросы теста | `id`, `test_id`, `school_id`, `type` (mcq/multi/open/matching), `content`, `max_score` |
| `question_options` | Варианты ответов | `id`, `question_id`, `content`, `is_correct`, `order` |
| `question_images` | Изображения к вопросам | `id`, `question_id`, `file_url`, `file_size` |
| `test_assignments` | Назначение тестов | `id`, `test_id`, `class_id`, `student_id` (null = весь класс), `deadline`, `school_id` |
| `test_sessions` | Сессии прохождения | `id`, `test_id`, `student_id`, `school_id`, `started_at`, `completed_at`, `status`, `late_submission` |
| `answers` | Ответы ученика | `id`, `session_id`, `question_id`, `student_id`, `school_id`, `response`, `score`, `created_at` |
| `test_results` | Итоговые результаты | `id`, `session_id`, `student_id`, `school_id`, `test_id`, `total_score`, `max_score`, `percent` |

> **Критично:** `school_id` присутствует во ВСЕХ таблицах + RLS в PostgreSQL.  
> Таблица `answers` партиционирована по `(school_id, month)`.  
> Индексы: `answers(student_id, created_at)`, `answers(test_id, created_at)`, `answers(school_id, subject_id)`.

## Граничные случаи

### GC-1: Ученик закрыл браузер в середине теста
- **Поведение:** Каждый ответ сохраняется немедленно (`auto-save` при выборе ответа)
- При повторном открытии: сессия восстанавливается с последнего сохранённого состояния
- Таймер продолжает идти на сервере (не на клиенте) — нельзя «заморозить» время
- Если дедлайн прошёл: сессия автоматически завершается с текущими ответами, статус `auto_completed`

### GC-2: Закончился таймер НТТ-симулятора
- Сервер принудительно закрывает сессию через background job
- Отправляются все ответы, сохранённые к моменту истечения времени
- Несохранённый последний ответ (если ученик не успел нажать) — **не засчитывается**
- Ученик видит уведомление: «Время истекло. Ваш тест завершён с текущими ответами»
- Статус сессии: `completed` (не `expired`) — важно для аналитики

### GC-3: Нет интернета при отправке ответов (offline mode)
- Service Worker перехватывает запрос, сохраняет в IndexedDB очередь `sync_queue`
- Background Sync API отправляет ответы при восстановлении соединения
- Если тест уже закончился по дедлайну: ответы принимаются с меткой `late_submission = true`
- Учитель видит в аналитике: N ответов с пометкой «offline sync»

### GC-4: Два устройства одновременно
- Сессия создаётся при нажатии «Начать» и привязана к `session_id`
- При попытке открыть тест на втором устройстве: предупреждение «Тест уже открыт на другом устройстве»
- Разрешается продолжить на втором устройстве (сессия переносится), но предыдущие несохранённые ответы теряются
- Это логируется в `audit_log` как `session_device_switch`

### GC-5: Ученик пытается пройти тест дважды
- Если `allowed_attempts = 1`: кнопка «Начать» заменена на «Просмотреть результат»
- Если `allowed_attempts > 1`: показывается «Попытка 2 из N», предыдущий результат виден
- Зачётный результат: лучший из попыток (по умолчанию) или последний (по настройке)

### GC-6: Ученик не авторизован в Telegram
- Redirect на страницу входа Zedly (login + пароль)
- Если это первый вход по OTP: обязательная смена пароля до доступа к тесту
- Если Telegram подключён в профиле: можно использовать «Войти через Telegram»
- После авторизации — возврат к тесту (сессия не создаётся до авторизации)

### GC-7: Учитель редактирует тест после назначения
- Если ни один ученик не начал: изменения применяются
- Если хотя бы один ученик начал: редактирование заблокировано, показывается предупреждение
- Учитель может создать новую версию теста (clone)

### GC-8: Вопрос с изображением — файл > 5 MB
- Форма показывает ошибку при загрузке: «Файл слишком большой. Максимум 5 MB»
- Загрузка не начинается (client-side validation)
- Дополнительная валидация на сервере (400 Bad Request)

### GC-9: Межшкольный челлендж — один из директоров отказывается
- Приглашение активно 72 часа
- Если школа B не приняла: статус `declined`, школа A получает уведомление
- Нельзя начать челлендж в одностороннем порядке

### GC-10: Offline — ученик синхронизировал ответы, но дедлайн давно прошёл
- Система принимает ответы с меткой `late_submission = true`
- Учитель получает уведомление в дашборде
- Учитель может принять или отклонить `late_submission` (настраивается на уровне теста)
- По умолчанию: принимаются (особенно для региональных школ с нестабильным интернетом)

## Acceptance Criteria

### AC-1: Создание теста
```gherkin
Given: Учитель авторизован
When: Он создаёт тест с 5 MCQ-вопросами и назначает классу
Then: Тест появляется в списке «Активные тесты»
And: Ученики класса получают уведомление через Telegram-бота
And: Тест доступен для прохождения немедленно
```

### AC-2: Прохождение теста (MCQ)
```gherkin
Given: Ученик авторизован и тест назначен его классу
When: Ученик открывает тест и отвечает на все вопросы
Then: Результат отображается сразу после завершения
And: Учитель видит результат в дашборде в течение 15 секунд
And: Карточка-результат доступна для шейринга
```

### AC-3: НТТ-симулятор — истечение времени
```gherkin
Given: Ученик проходит НТТ-симулятор, ответил на 45 из 60 вопросов
When: Истекают 90 минут
Then: Система автоматически завершает сессию
And: Засчитываются ответы на 45 вопросов
And: Ученик видит результат с разбором ошибок по темам
And: Статус сессии — completed (не expired)
```

### AC-4: Offline-синхронизация
```gherkin
Given: Ученик скачал тест и проходит его без интернета
When: Интернет восстанавливается
Then: Ответы автоматически синхронизируются в течение 30 секунд
And: Учитель видит результат в дашборде
And: Синхронизированные ответы помечены offline_sync = true
```

### AC-5: Изоляция данных
```gherkin
Given: Учитель школы A авторизован
When: Он пытается получить доступ к тестам школы B (через API или URL)
Then: Система возвращает 403 Forbidden
And: Попытка записывается в audit_log
And: Учитель не получает информации о существовании ресурса школы B
```

### AC-6: Два устройства
```gherkin
Given: Ученик начал тест на телефоне
When: Он открывает тот же тест на компьютере
Then: Система показывает предупреждение «Тест открыт на другом устройстве»
And: Ученик может продолжить на текущем устройстве
And: Событие device_switch записывается в audit_log
```

### AC-7: Открытый ответ — ожидание проверки
```gherkin
Given: Тест содержит вопрос с открытым ответом
When: Ученик завершает тест
Then: Статус результата — awaiting_review
And: Учитель видит уведомление «1 ответ требует проверки»
And: Итоговый балл не отображается ученику до проверки учителем
```

### AC-8: Межшкольный челлендж — 48 часов
```gherkin
Given: Директор школы A отправил челлендж школе B
When: Школа B принимает вызов
Then: Оба класса получают тест с единым дедлайном 48 часов
And: Результаты видны на публичной странице челленджа (без персональных данных)
And: По истечении 48 часов публикуется итоговая карточка-результат
```

## Производительность и KPI

| Метрика | Цель | Критический порог |
|---------|------|------------------|
| Время создания теста (5 MCQ вручную) | ≤ 3 мин | > 7 мин = проблема UX |
| Время загрузки теста учеником | ≤ 1.5 сек | > 3 сек = потеря ученика |
| Процент завершения теста (completion rate) | ≥ 85% | < 70% = проблема с UX/таймером |
| Время автосохранения ответа | ≤ 300 мс | > 1 сек = риск потери ответа |
| Время синхронизации offline-ответов | ≤ 30 сек | > 2 мин = плохой UX |
| Одновременных сессий (100 школ) | 10,000 | — |
| Время генерации результата после завершения | ≤ 2 сек | > 5 сек = проблема |

## Архитектурные заметки

- **Таймер на сервере**, не на клиенте. Клиент отображает countdown, сервер принудительно закрывает сессию. Это предотвращает читерство через DevTools.
- **Auto-save каждого ответа**: `POST /api/sessions/{id}/answers` после каждого выбора ответа (не только при завершении).
- **Идемпотентность**: повторная отправка того же ответа не создаёт дубликат (upsert по `session_id + question_id`).
- **Background job** (Bull/Celery): каждые 60 секунд проверяет истёкшие сессии и принудительно закрывает.
- **analytics_snapshots**: результаты агрегируются в `analytics_snapshots` каждые 15 минут. Дашборд учителя читает из снапшотов, не из raw `answers`.
