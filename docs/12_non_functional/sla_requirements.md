## 1. Uptime и доступность

### Целевые показатели

| Этап | Uptime | Допустимый downtime/месяц | Допустимый downtime/год |
|------|--------|--------------------------|------------------------|
| MVP | 99.5% | 3 ч 39 мин | 43 ч 48 мин |
| Госконтракты | 99.9% | 43 мин | 8 ч 46 мин |

### Плановые технические обслуживания

- Разрешённое окно: **01:00–05:00 (UTC+5)** — минимальная активность пользователей.
- Плановые обслуживания **не засчитываются** в downtime при условии предварительного уведомления (не менее 24 часов).
- Уведомление: Telegram-канал для администраторов школ + статус-страница.

### Измерение uptime

- Проверка доступности: synthetic monitoring с интервалом 1 минута (внешний monitor).
- Инцидент считается открытым, если сервис недоступен более 2 минут подряд.
- Отчёт по uptime публикуется ежемесячно на статус-странице.

## 2. Производительность

### Целевые latency

| Операция | P50 | P95 | Тип |
|----------|-----|-----|-----|
| Загрузка дашборда | — | < 1 000 мс | Синхронный |
| Сабмит ответа теста | — | < 500 мс | Синхронный |
| API latency (общий) | < 200 мс | < 1 000 мс | Синхронный |
| Генерация отчёта (PDF) | — | < 30 сек | Асинхронный |

### Условия измерения

- Метрики замеряются в **production-окружении** при штатной нагрузке.
- P95 — 95-й перцентиль времени ответа за скользящее окно 5 минут.
- Генерация PDF — время от постановки задачи в очередь до готовности файла.
- Метрики собираются через APM (Application Performance Monitoring); алерт при превышении порога P95 более чем на 20% в течение 5 минут.

### Бюджет деградации

При нагрузке выше 80% от пикового значения (см. раздел [4](#4-одновременные-пользователи)) допускается деградация latency не более чем в 2× от целевых P95 значений.

## 3. Масштабируемость

### Инфраструктурные этапы

| Этап | Школы | Инфраструктура | Ориентировочная стоимость/мес |
|------|-------|----------------|-------------------------------|
| MVP | до 10 | Один VPS | ~$30 |
| Growth | до 100 | Горизонтальный scale-out + PostgreSQL read replica | $200–300 |
| Scale | до 1 000 | Шардирование по `school_id`, Kubernetes или managed PaaS | $1 500–2 000 |

### Принципы масштабирования

**MVP (до 10 школ):** монолитное развёртывание на одном VPS; вертикальное масштабирование при необходимости.

**Growth (до 100 школ):** горизонтальное масштабирование stateless application-серверов за load balancer; read replica PostgreSQL для аналитических запросов и генерации отчётов.

**Scale (до 1 000 школ):** шардирование базы данных по `school_id` — каждый шард обслуживает диапазон школ. Оркестрация через Kubernetes или managed PaaS (Railway, Render, или облачный провайдер в РУз). CDN для статических ассетов.

### Триггеры для перехода между этапами

| Переход | Триггер |
|---------|---------|
| MVP → Growth | CPU > 70% sustained или latency P95 > 1.5× цели в течение 7 дней |
| Growth → Scale | 80+ школ подключено или 4 000+ одновременных сессий |

## 4. Одновременные пользователи

### Пиковая нагрузка

Пик активности приходится на **08:00–09:00 (UTC+5)** — начало учебного дня, массовый вход учеников и учителей.

### Целевые показатели по этапам

| Этап | Одновременных сессий | Одновременных сессий (пик ×1.5) |
|------|---------------------|----------------------------------|
| MVP (до 10 школ) | 500 | 750 |
| Growth (до 100 школ) | 5 000 | 7 500 |

### Нагрузочное тестирование

- Нагрузочный тест проводится перед каждым крупным релизом.
- Целевой сценарий: симуляция пикового трафика (×1.5 от расчётного значения) в течение 15 минут.
- Критерий прохождения: latency P95 остаётся в пределах целевых значений (раздел [2](#2-производительность)); error rate < 0.1%.

## 5. Offline поддержка

### Требования

- Прохождение теста должно работать **без активного интернет-соединения** (PWA offline-first).
- Данные теста сохраняются локально (IndexedDB / Service Worker cache).
- При восстановлении соединения результаты **синхронизируются автоматически** без участия пользователя.

### Поведение при конфликтах синхронизации

- Приоритет: локальные данные ученика не перезаписываются серверными.
- Конфликт (один и тот же тест сдан дважды — offline и online): сохраняется первая по timestamp запись; дублирующая помечается флагом `sync_conflict` для ручной проверки.

### Ограничения offline-режима

| Функция | Offline | Online |
|---------|---------|--------|
| Прохождение теста | ✅ | ✅ |
| Просмотр результатов | ✅ (кэш) | ✅ |
| Дашборд ученика | ✅ (кэш) | ✅ |
| Генерация PDF-отчёта | ❌ | ✅ |
| Telegram-уведомления | ❌ | ✅ |
| Создание новых тестов (учитель) | ❌ | ✅ |

## 6. Поддерживаемые платформы

### Браузеры

| Браузер | Минимальная версия | Приоритет |
|---------|--------------------|-----------|
| Chrome (Desktop) | 90+ | Основной |
| Chrome (Android) | 90+ | Основной |
| Firefox | 88+ | Основной |
| Safari (iOS/macOS) | 14+ | Основной |
| Samsung Internet | 14+ | Дополнительный |
| Edge | 90+ | Дополнительный |

### Мобильные и альтернативные интерфейсы

| Платформа | Тип | Статус |
|-----------|-----|--------|
| Адаптивный web (responsive) | PWA | ✅ MVP |
| Telegram Mini App | Web App API | ✅ MVP |
| Нативное iOS-приложение | Native | Вне скоупа MVP |
| Нативное Android-приложение | Native | Вне скоупа MVP |

### Требования к сети

- **Минимальная скорость:** 2G (~50–250 kbps) — для базового прохождения теста.
- **Рекомендуемая скорость:** 3G и выше — для полного функционала включая загрузку медиа.
- Все ассеты оптимизированы для медленных соединений: lazy loading, сжатие изображений (WebP), минификация JS/CSS.
- Telegram Mini App дополнительно снижает требования к сети за счёт нативного кэширования клиента Telegram.

### Разрешения экрана

- Минимальное поддерживаемое разрешение: **320px** (ширина viewport).
- Интерфейс тестирования оптимизирован для экранов 360–414px (типичные Android-устройства в РУз).
